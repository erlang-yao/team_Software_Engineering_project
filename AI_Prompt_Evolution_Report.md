# AI 辅助测试的 Prompt 演化实验报告

## 1. 实验背景

本次实验中，我围绕“如何用 AI 辅助完成核心 API 模块的单元测试设计与实现”开展了一次 Prompt 演化实践。实验目标并不只是让 AI 生成一份“看起来有测试”的代码，而是观察在 Prompt 从粗糙到结构化的过程中，测试结果在可编译性、覆盖深度、断言质量和可维护性上会发生怎样的变化。

从项目背景来看，我当前维护的是一个 C++ 控制台宝可梦风格 MUD 游戏，核心模块包括战斗、玩家、地图、队伍管理和物品系统。由于课程要求不仅要完成测试，还要体现 Prompt 工程方法，因此我把本次任务拆成了一个比较明确的实验链条：

1. 先给出较粗糙的初始 Prompt。
2. 观察 AI 输出中暴露的问题。
3. 根据问题重新设计更结构化的 Prompt。
4. 比较优化前后结果差异，并总结哪些 Prompt 工程策略真正有效。

换句话说，这次实验的重点不是“AI 帮我写测试”，而是“我如何通过 Prompt 设计，把 AI 输出逐步引导到更可靠、更贴合项目代码的方向上”。这也是我理解的实验意义所在。

## 2. 实验对象说明

本次实验我选取了两组模块作为对象，分别对应两类不同的测试场景。

### 2.1 案例一：战斗规则相关测试

案例一聚焦于战斗规则相关逻辑，主要包括：

- 伤害计算
- 逃跑概率
- 捕捉概率

之所以选择这一组，是因为 `battle.cpp` 中原始逻辑与 `std::cin`、`std::cout` 以及回合制流程耦合较强，像 `Battle::run()`、菜单选择、捕捉流程、切换精灵等都直接混在交互过程中。如果一开始就要求 AI 对 `battle.cpp` 整体写单元测试，很容易生成难以编译、难以隔离、难以维护的测试。

因此我在实验过程中新增了 `battle_rules.h/.cpp`，把伤害计算、逃跑概率、捕捉概率抽取为纯规则函数，作为首批测试目标。这一步本身也是“为了提高可测试性而进行代码抽取”的一个实验结论。

### 2.2 案例二：玩家物品与队伍管理测试

案例二聚焦于 `player.cpp` 中较适合单元测试的公开接口，包括：

- `useItemOnPokemon`
- `revivePokemon`
- `movePokemonToStorage`
- `movePokemonFromStorage`

我选择这一组的原因是，`player.cpp` 的公开接口相对清晰，输入输出边界明确，状态变化集中在队伍、仓库、HP 和物品数量上，天然适合做高质量断言。而且这些接口存在比较丰富的边界情况，例如：

- `itemIndex` 越界
- `pokemonIndex` 越界
- `item count = 0`
- revive 药误用
- fainted 宝可梦不能使用治疗药
- 队伍只剩 1 只不能存仓
- 队伍满 6 只不能从仓库取回

这类场景很适合观察 Prompt 优化前后，AI 是否真的能从“只测 happy path”提升到“有边界意识、有状态一致性意识”的测试设计层面。

## 3. Prompt 演化案例一：战斗规则测试

### 3.1 初始 Prompt

我最开始使用的 Prompt 非常简单：

> 请为 battle.cpp 写 C++ 单元测试，覆盖 calculateDamage、tryEscape、catchPokemon。

从当时的想法来看，这种 Prompt 的优点是直接、成本低，但问题也恰恰出在“过于直接”。它只告诉 AI“测什么函数”，却没有说明这些函数的可访问性、是否适合作为测试入口、测试应该如何断言，也没有约束输出代码是否必须可编译。

### 3.2 AI 初始输出存在的问题

拿到第一轮结果后，我发现它更像“机械补全”而不是“测试设计”。主要问题有以下几类：

- 直接尝试测试 `Battle` 的私有函数，忽略了封装边界。
- 测试代码可能无法编译，因为它默认可以直接访问内部状态或私有逻辑。
- 倾向于在测试中复制实现公式，再用这个公式去断言结果，形成重言式测试。
- 边界覆盖不足，没有主动考虑最小/最大逃跑概率、大师球特殊逻辑、属性无效场景等。
- 没有识别出 `battle.cpp` 与控制台交互和回合流程强耦合，不适合直接作为首批单元测试目标。

从这个结果我意识到，问题不完全在 AI，而在于我给出的任务描述没有提供“测试策略层”的约束。AI 按照最表面的方式理解了需求，于是也输出了最表面的结果。

### 3.3 前后对比表

| 对比项 | 初始 Prompt 输出 | 改进后 Prompt 输出 |
|---|---|---|
| 测试对象 | 直接面向 `battle.cpp` 内部成员 | 转向 `battle_rules.cpp` 纯规则函数 |
| 可编译性 | 容易触碰私有函数或依赖交互状态，编译风险高 | 纯函数接口明确，编译和组织更稳定 |
| 断言质量 | 容易复制公式做断言，测试价值有限 | 以业务行为断言为主，避免重言式测试 |
| 边界覆盖 | 容易遗漏上下限、特殊分支和属性无效 | 能覆盖基础值、上下限、特殊球、免疫场景 |
| 结果可维护性 | 强依赖实现细节，后续修改容易连带失效 | 依赖规则行为，维护成本更低 |

### 3.4 改进后的 Prompt

在第二轮中，我不再只给一个功能点，而是把 Prompt 改写成带有角色、步骤、约束和目标范围的结构化描述。核心思路是：先让 AI 理解“什么是合适的测试对象”，再让它生成代码。

我最终使用的改进版 Prompt 具备以下几种 Prompt 工程策略。

#### 3.4.1 结构化指令

我把要求拆成“测试对象范围”“输出顺序”“禁止事项”“覆盖要求”几个部分，而不是一句话提需求。这样做的作用是减少 AI 误解任务边界的概率。例如我明确要求它不要直接围绕 `Battle::run()` 这种交互流程下手，而要基于 `battle_rules.cpp` 的纯函数来写测试。

#### 3.4.2 Few-shot 思路

虽然我没有单独给出完整的示例代码块，但我在 Prompt 中通过列举具体目标场景，等于给了一个“小样本的期望风格”，例如“最小逃跑率”“最大逃跑率”“大师球特殊值”“属性无效场景”等。这种列举本质上起到了 few-shot 的引导作用，让 AI 更容易朝“按测试点组织代码”的方式输出，而不是只给一组泛泛的示例。

#### 3.4.3 CoT（分步推理）

我要求 AI “先列测试点，再写代码”，这其实是在强制它先进行测试设计，再进入实现阶段。对我来说，这一步非常关键，因为如果不要求分步推理，AI 很容易直接跳到代码生成，结果往往缺少完整覆盖。先列点相当于把它的思路外显化，也方便我检查有没有漏测边界。

#### 3.4.4 角色扮演

我把角色设定为“资深 C++ 测试工程师”。角色扮演的作用不是增加语气，而是让 AI 在输出时更偏向测试设计视角，主动考虑可测试性、边界情况和代码组织，而不是只从“能写几行测试代码”的角度完成任务。

### 3.5 改进后的 Prompt 原文

> 你现在扮演一名资深 C++ 测试工程师。请不要直接测试 `Battle::run()` 这类与控制台输入耦合的流程，而是基于 `battle_rules.cpp` 中的纯函数编写 doctest 风格单元测试。先列出测试点，再生成代码。测试必须覆盖正常路径、边界路径、特殊路径。禁止写重言式测试，不要在测试里完整复制实现公式。请重点覆盖：最小/最大逃跑概率、同速基础概率、大师球特殊逻辑、捕捉概率上下限、最小伤害、本系加成、属性克制与属性无效场景。

### 3.6 最终测试结果总结

改进后的结果明显更贴近真正可用的单元测试。一方面，AI 不再执着于直接操作 `Battle` 类内部逻辑，而是接受了“先抽规则、再测规则”的思路；另一方面，测试断言也从“复制公式比结果”转向“验证业务行为”，例如：

- 同速时逃跑率应返回基础概率。
- 速度差极大时应被钳制到最大值或最小值。
- 大师球应返回固定特殊捕捉率。
- 本系技能伤害应高于同威力的非本系技能。
- 克制伤害应高于被抵抗伤害。
- 电属性对地面属性无效时应返回 0 伤害。

最终我也在项目中实际新增了 `battle_rules.h/.cpp`，并围绕这三个纯函数完成了测试。这说明 Prompt 优化不只是“把答案写得更漂亮”，而是真正改变了测试对象的选择方式和测试设计质量。

## 4. Prompt 演化案例二：玩家物品与队伍管理测试

### 4.1 初始 Prompt

案例二的起点同样比较粗糙，我最开始只写了一句：

> 请为 `Player::useItemOnPokemon` 写单元测试。

这类 Prompt 的问题是看似明确，实际上只指定了一个函数名，没有指定覆盖深度、失败场景、状态一致性要求，也没有要求测试风格和输出步骤。

### 4.2 AI 初始输出存在的问题

第一轮结果的主要问题也比较典型：

- 只覆盖 happy path，例如“治疗药使用后 HP 增加”。
- 忽略 `itemIndex` 越界、`pokemonIndex` 越界、`item count = 0` 等输入边界。
- 忽略 revive 药物限制，没有区分 `useItemOnPokemon` 和 `revivePokemon` 的职责边界。
- 忽略 fainted 宝可梦不能使用治疗药水。
- 忽略仓库存取相关的边界条件，如队伍只剩 1 只不能存仓、队伍满 6 只不能取回。
- 对失败场景往往只断言返回值，没有验证“状态不应被污染”。

这一轮结果让我意识到，如果不明确要求“失败后状态保持不变”，AI 非常容易生成只看返回值的浅层测试，但这类测试很难发现真正的状态错误。

### 4.3 前后对比表

| 对比项 | 初始 Prompt 输出 | 改进后 Prompt 输出 |
|---|---|---|
| 测试对象 | 只盯住 `useItemOnPokemon` 单一成功路径 | 扩展到 `useItemOnPokemon`、`revivePokemon`、仓库存取等接口 |
| 可编译性 | 通常能写出代码，但组织较松散 | 按 doctest 风格组织，更接近实际测试文件 |
| 断言质量 | 主要看返回值或 HP 变化 | 同时检查物品数量、队伍大小、HP 与经验状态 |
| 边界覆盖 | 只测常规成功流程 | 明确覆盖索引越界、数量为 0、误用、队伍上下限 |
| 结果可维护性 | 依赖单一路径，回归价值有限 | 面向接口行为和状态一致性，回归价值更高 |

### 4.4 改进后的 Prompt

第二轮中，我把 Prompt 改写成更像一份测试设计任务书，而不是简单命令。目的就是让 AI 从“函数调用者”切换到“测试架构设计者”的视角。

这里我同样使用了几种 Prompt 工程策略。

#### 4.4.1 结构化指令

我明确指定测试对象集合、必须覆盖的边界、失败场景的状态检查方式，以及最后输出的代码风格。这样做的好处是让 AI 不再只围绕单个接口自由发挥，而是在统一框架内组织测试。

#### 4.4.2 Few-shot 思路

我在 Prompt 中显式列出具体边界情形，例如“复活药误用”“fainted 宝可梦不可治疗”“队伍满 6 只不能取回”等。虽然不是完整示例代码，但这些高质量样例点本身就构成了 few-shot 风格的引导，让 AI 更容易理解“我想要的不是普通演示，而是带业务边界的测试”。

#### 4.4.3 CoT（分步推理）

我要求“先用表格列出测试点，再生成 doctest 风格代码”。这种分步方式迫使 AI 先做输入空间拆分，再决定如何写断言。对复杂接口来说，这一步尤其重要，因为很多 bug 并不来自正常路径，而来自失败路径和状态残留。

#### 4.4.4 角色扮演

我把角色设为“测试架构师”。这样可以让 AI 的输出从“局部写几个测试”提升到“先规划覆盖面，再实现测试代码”的层次，更符合课程实验中对测试设计能力的要求。

### 4.5 改进后的 Prompt 原文

> 你现在扮演一名测试架构师，请为 `Player` 模块设计 doctest 风格单元测试。先用表格列出测试点，再写代码。必须覆盖 `useItemOnPokemon`、`revivePokemon`、`movePokemonToStorage`、`movePokemonFromStorage`，必要时补充 `swapTeamPokemon`。强制覆盖边界条件，包括索引越界、物品数量为 0、复活药误用、昏迷宝可梦不可治疗、队伍最少 1 只、队伍最多 6 只。所有失败场景都要额外断言状态不变，例如物品数量不减少、队伍大小不变化、HP 不变化。最后输出实际可编译风格的 doctest 测试代码。

### 4.6 最终测试结果总结

改进后的输出相比第一轮完整得多，主要体现在两个层面。

第一，测试范围从单一函数扩展到了多个公开接口，能够更真实地覆盖 `player.cpp` 这一模块的核心行为。第二，断言方式从“调用成功没有报错”升级成“返回值 + 状态一致性”的组合断言。比如：

- 对越界输入，除了断言返回 `false`，还检查物品数量和 HP 不变。
- 对 revive 药误用，检查计数不减少、目标状态不变化。
- 对 fainted 宝可梦使用治疗药，验证调用失败且不会错误回血。
- 对经验药水，除了断言使用成功，还检查 `level`、`exp`、`maxExp` 和能力值变化。
- 对仓库存取，检查队伍大小和仓库大小是否符合边界约束。

这类结果已经不只是“能跑的测试”，而是开始具备回归测试价值和维护价值。

## 5. 覆盖率设计与达成思路

### 5.1 覆盖率目标

本次实验中，我给自己设定的覆盖率目标是：核心 API 模块测试覆盖率达到 **80% 及以上**。

这里我特别强调“核心 API 模块”，而不是整个项目整体覆盖率。原因是本项目是一个控制台回合制程序，不同模块的可测试性差异很大，如果一开始就把所有模块纳入统一目标，反而会把实验重点冲淡。

### 5.2 本次覆盖范围定义

本次覆盖范围只定义为以下两个文件：

| 覆盖模块 | 纳入原因 |
|---|---|
| `player.cpp` | 公开接口清晰，状态变化集中，适合做边界与状态一致性测试 |
| `battle_rules.cpp` | 由 `battle.cpp` 中抽取出的纯规则函数，可做确定性单元测试 |

这一定义是我在实验过程中的主动选择，而不是回避困难。我的判断标准是：首批覆盖对象必须同时满足“接口稳定”“输入输出清楚”“无需依赖控制台交互”“断言可确定”这几个条件。

### 5.3 为什么不把其他模块纳入首批重点

本次我没有把 `game.cpp`、`Battle::run()`、`TeamView::run()` 作为首批覆盖重点，主要原因如下：

- 它们与控制台 I/O 强耦合，直接依赖 `std::cin`、`std::cout`。
- 流程逻辑与菜单交互混在一起，测试入口不够干净。
- 即使可以勉强测试，编写和维护成本也会比较高。
- 这类测试往往更脆弱，稍微调整输出文本或交互顺序就可能导致失败。

因此，在没有进一步重构输入输出适配层之前，我认为把它们纳入首批覆盖重点并不划算。先把规则层和公共接口层测扎实，再考虑流程层测试，会更符合工程实践。

### 5.4 达成思路说明

由于本次实验的重点在于测试对象选择与 Prompt 演化，所以我在覆盖率统计时坚持只统计本次定义的核心范围，而不是把所有源码混在一起计算。最终我实际运行了两组测试文件：

- `tests/test_player.cpp`
- `tests/test_battle_rules.cpp`

两组测试合并执行结果为：

- `28` 个 `TEST_CASE`
- `108` 条断言
- 全部通过，无失败用例

在覆盖率统计方面，我最终得到的核心范围结果如下：

| 统计范围 | lines covered | lines valid | line-rate |
|---|---:|---:|---:|
| `player.cpp` + `battle_rules.cpp` | 151 | 157 | 96.18% |

这说明本次实验不仅在设计层面把覆盖范围限定清楚了，而且在实际执行结果上也达到了“核心 API 模块覆盖率不低于 80%”的预期目标。

另外，本次统计过程中我还遇到了一个比较真实的工程问题：覆盖率工具在当前中文路径工程目录下导出 Cobertura XML 时，文件尾部会被截断。因此我最后采用了一个更稳妥的处理方式：先在纯英文临时路径下完成覆盖率导出，再把完整 XML 结果复制回当前项目目录保存。这个过程也让我意识到，实验报告中的“工具结果”并不只是数字本身，还包括对工具限制和工程环境问题的识别与处理。

综合来看，我认为这种覆盖范围定义和达成方式更真实，也更符合课程实验对“设计合理性”和“过程完整性”的要求。

## 6. 实验总结与个人反思

通过这次实验，我对 Prompt 工程和单元测试设计之间的关系有了更直观的认识。最明显的一点是，Prompt 质量会直接影响测试质量。初始 Prompt 如果只写“帮我测某个函数”，AI 往往就只会给出最表层、最直接的结果；而当我把角色、步骤、边界、禁止事项和输出形式说清楚以后，AI 的输出质量会明显提升。

从案例一来看，我最大的收获是“测试对象的选择比直接写测试更重要”。如果我一开始坚持围绕 `battle.cpp` 的交互流程做单元测试，很可能会陷入难编译、难维护、难断言的问题。相反，把其中适合测试的伤害计算、逃跑概率、捕捉概率抽取到 `battle_rules.h/.cpp` 后，测试工作一下子变得清晰很多。这让我认识到，AI 辅助测试并不只是生成代码，更可能反过来推动代码结构朝着更可测试的方向改进。

从案例二来看，我体会到“边界条件必须被写进 Prompt”。像 `itemIndex` 越界、`pokemonIndex` 越界、物品数量为 0、revive 药误用、昏迷宝可梦不能治疗、队伍上下限等问题，如果我不在 Prompt 中明确指出，AI 通常不会主动把这些场景全部补齐。也就是说，AI 能放大测试设计质量，但前提是我自己先把测试设计想清楚。

另外，我也进一步理解了结构化指令、Few-shot、CoT 和角色扮演这几种方法在实际任务中的区别。它们不是抽象概念，而是能直接影响输出质量的具体策略：

- 结构化指令帮助我限定边界，减少跑题。
- Few-shot 式的测试点列举帮助我传达期望结果风格。
- CoT 式的“先列点再写代码”帮助我检查覆盖面。
- 角色扮演帮助 AI 以测试设计者而不是普通代码补全器的视角工作。

综合来看，我认为这次实验最重要的结论不是“AI 可以帮我写测试”，而是“Prompt 工程本身会决定 AI 测试结果的上限”。粗糙 Prompt 得到的往往只是表面覆盖，结构化 Prompt 才更容易得到真正有回归价值的测试。

结合这次最终结果来看，我也更加确认了一点：高质量 Prompt 不只是让 AI 输出“更多代码”，而是能让测试工作从一开始就沿着正确方向展开。案例一中，如果没有先把 `battle.cpp` 中可抽取的纯规则逻辑独立出来，我很难在后续得到高覆盖率、低脆弱性的测试结果；案例二中，如果没有在 Prompt 里明确写出失败场景和状态不污染要求，即使最后代码能运行，测试质量也很难支撑后续回归。换句话说，Prompt 设计实际上影响了测试对象选择、断言策略、覆盖率达成路径，甚至影响了我是否需要先改造代码再做测试。

从课程实验的角度看，这次实践也让我明白，AI 辅助开发最有价值的地方，不是替我跳过思考，而是把“原本需要我自己模糊摸索”的过程，转化成“我通过更精确的指令去驱动结果”的过程。真正有效的不是一句“帮我写测试”，而是我能否把测试目标、约束、优先级、边界条件和输出标准表达清楚。

### 6.1 下一步改进方向

如果继续完善这个项目，我下一步会从三个方向推进。

首先，我会继续拆分 `battle` 中与控制台交互强耦合的流程逻辑，把更多“流程内部规则”和“交互展示”分离开来。这样做的目的，是让测试入口不只停留在规则层，而能逐步延伸到更高层的战斗流程。

其次，我希望让测试入口从“规则可测”逐步扩展到“流程可测”。当前 `battle_rules.cpp` 已经证明规则层测试是可行的，后续如果能把回合推进、动作结算、输入决策等进一步拆分，就有机会为更完整的战斗流程建立稳定测试。

最后，我会把这次验证有效的 Prompt 模板沉淀成团队可复用的规范，比如“先列测试点再写代码”“失败场景必须检查状态不污染”“交互耦合模块优先抽纯函数”等。这样不仅能提升我个人后续使用 AI 的效率，也能让团队在协作时形成更一致的测试生成标准。

如果后续继续扩展，我还希望把这次已经验证有效的测试与覆盖率流程进一步标准化，例如固定采用 UTF-8 编译参数、固定在英文路径下导出覆盖率结果、固定保留核心范围统计口径。这样一来，后面的实验结果就会更稳定，也更方便团队成员之间复现。
